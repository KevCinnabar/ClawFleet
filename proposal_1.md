# 目标

构建一个面向 AI Agent 协作的“Docker 式”运行与交付层：将 OpenClaw 及其生态的安装、配置、部署、连接与运行标准化为一份可读、可校验、可版本化的 manifest，使用户能在本地或云端一键启动一组角色化 Agent（PM/架构/开发/测试），并在不同环境间稳定迁移。

该项目的核心目的包括：

- 降低门槛：把 bash/k8s/依赖配置等隐性复杂度封装起来，让非资深工程师也能使用
- 保证状态：实现记忆/日志/产物的持久化与隔离，重启与扩缩容不丢上下文
- 加速协作：提供可插拔的协作流与标准化 Prompt/Skill 库，让团队能复制最佳实践并持续迭代
- 可治理可扩展：对模型、工具、成本与观测提供统一的控制面能力，便于规模化与长期维护

## 目标用户 & 使用场景

- 独立开发者/小团队：需要快速搭建 PM/架构/开发/测试协作流
- 企业内部 PoC：需要在不同环境（本地、ECS、EKS/K8s）快速复现
- 需要可控成本与可观测性：模型可切换、用量可计费、运行可追踪

## 关键价值（Why Now）

- 更低门槛：不要求用户熟悉 bash/k8s 细节
- 可迁移：同一份配置在不同云/不同机器可重复部署
- 可持续：记忆与数据持久化，Pod 重启不丢状态
- 可扩展：标准化 Prompt/Skill/工具接入，形成生态

## 范围（In Scope / Out of Scope）

### In Scope

- Agent Group 运行时封装与部署（本地 + 云端）
- 配置清单（manifest）与参数体系
- 记忆/状态持久化与隔离
- 第三方工具连接的“声明式”接入（例如 GitHub/Notion/飞书等）
- Prompts / Skills 的标准库与版本管理

### Out of Scope（暂不做）

- 自研大模型训练
- 完整的企业级权限/审计体系（可先提供基础能力）
- 复杂 UI（可先 CLI + Web 控制台最小可用）

## 角色（Agent Personas）

- PM：拆解需求、规划迭代、输出 PRD/任务
- 架构师：系统设计、技术选型、接口规范
- Developer：实现功能、联调、提交 PR
- Tester：测试策略、用例、回归、质量门禁

## 配置与参数（Manifest Spec）

### 需要可配置的核心参数

1. LLM（Provider/Model/Context/温度/限流/重试）
2. 通讯工具（IM、Webhooks、邮件等）
3. Agent（角色、权限、工具、skill、prompt 版本）
4. 运行资源（CPU/GPU/内存、并发、超时、预算上限）
5. 数据与存储（记忆存储后端、向量库、日志、对象存储）
6. 观测（Tracing/metrics/logging、Prompt/Tool 调用记录）

### 配置清单可读性要求

- 面向非程序员：字段命名清晰、示例完整、默认值合理
- 可校验：schema 校验 + 友好错误信息
- 可版本化：配置与 prompt/skill 绑定版本（可回滚）

## 当前可行方案的对比（Problems）

1. 仅使用 OpenClaw：需要使用者熟悉 bash commands，部署/扩展门槛高
2. OpenClaw 记忆只在实体机器：Pod 重启会导致记忆丢失
3. 仍需手动连接第三方程序：代码库、文档、IM 等接入成本高
4. OpenClaw 可配置项过多、自由度过高，非从业者上手成本大；需收敛为少量关键选项，用“选择”替代“编辑”。

## 需要实现的功能（MVP → v1）

### MVP（优先级最高）

1. 记忆分开管理：按 Agent / Project / Workspace 隔离
2. 简化配置清单：一份 manifest 即可启动（含示例与注释）
3. 部署不丢记忆：提供推荐的持久化方案（如 DB/对象存储/向量库）

### v1（增强）

1. Agent 合作流定制化：可插拔 workflow（例如：需求→设计→开发→测试）
2. 标准化 Prompts（核心竞争力）：Prompt/Skill 模板库 + 评测/回归

## 架构建议（待确认）

- 控制面（Control Plane）：管理配置、版本、连接、权限、审计/计费
- 数据面（Data Plane）：执行 Agent、调用工具、处理队列、运行 sandbox
- 存储层：
    - 记忆：KV/Doc + 向量库（可插拔）
    - 日志：结构化日志 + 可检索
    - 产物：对象存储（如 S3/OSS）

## 关键技术问题清单（Open Questions）

- 记忆持久化：选择什么存储后端？如何做多租户隔离？
- 安全与密钥：第三方 token 如何加密/轮换？
- 执行隔离：Agent 工具执行如何 sandbox？（容器/权限）
- 成本控制：按 token/调用次数/运行时长计费与预算上限如何做？
- Prompt 评测：如何建立基准集与回归机制？

## 里程碑（建议）

- M0：manifest 草案 + 本地一键启动（hello world）
- M1：记忆持久化 + Pod 重启不丢状态
- M2：GitHub/Notion/飞书之一的声明式连接
- M3：Prompt/Skill 标准库 + 基础评测

## 交付物（Definition of Done）

- 能通过一份 manifest 启动一个 Agent Group
- 支持至少 1 种云部署路径（例如 K8s）
- 记忆在重启后保持一致，并支持隔离策略
- 有最小可用的观测（日志 + 关键指标）
- 文档：快速开始、配置说明、常见问题
